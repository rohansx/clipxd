# clipxd — product overview

> *Record once. Humans watch it. Agents read it.*
> A fast, Loom-style screen recorder with cinematic capture — whose shared link is fully legible to an LLM. Transcript, on-screen text, UI events, and salient visual moments, all queryable from the URL. Owned recorder, local-first, powered underneath by the **veyo** visual-event codec.

**Product name:** `clipxd` (clip + index — the index is the product).
**Engine:** veyo (separate tool, sits beneath — see §2).
**License intent:** open-core. Apache-2.0 core + optional commercial hosted/compliance layer. (Clean-room recorder, not a port of AGPL code — see [licensing](licensing.md).)

---

## 1. The one thing this product is

A single combined recording tool that is genuinely better than Loom *and* better than Cap for one reason no one else delivers: **the recording it produces is readable and processable by an agent from just the URL.**

The recorder market is otherwise solved — Cap nailed instant + studio capture and cursor auto-zoom; Screen Studio nailed cinematic zoom, backgrounds, and device mockups. clipxd reimplements that capture + beautification layer as one fast, owned tool (Rust), and then does the part nobody does: turns every recording into a structured index an agent can query.

The product is two things fused:
1. a **thin, fast, beautiful recorder** (the Cap + Screen Studio experience, rebuilt and owned)
2. an **agent-legible index** generated by veyo and exposed over MCP (the differentiator)

The recorder gets you adoption. The index is the moat.

## 2. veyo is the engine, clipxd is the product

veyo is a separate local-first **visual-event codec**: it keeps a text world-state on-device and emits a compact delta only when the scene meaningfully changes, so visual data lives as cheap text instead of streaming pixels to a VLM. "An agent understands every frame" is economically impossible taken literally (~2.6B tokens/day captioning every 1080p frame). veyo's salience / habituation gate is the mechanism that decides **which moments are worth enriching** — that's what makes agent-legible video affordable.

clipxd is veyo's first user-facing surface. A clip is *a recorded session run through veyo, with the video file attached.* clipxd owns the recorder, the share layer, and the product UX; veyo owns the codec. They ship and version independently.

## 3. Architecture — owned thin recorder, fat index

```
        ┌──────────────────── clipxd recorder (OWNED, THIN — Rust) ─────────────────────┐
        │  capture: pixels + audio + cursor/click/key events                              │
        │  cinematic layer: auto-zoom, backgrounds, mockups   (Screen Studio-class, owned) │
        │  browser mode: DOM / a11y / console / network trace  (rrweb/Playwright-style)    │
        └──────────────────────────────────────────┬───────────────────────────────────────┘
                                                    │  raw session (video + event track)
                                                    ▼
                                          veyo-core  (salience gate, CPU, no imagery leaves device)
                                                    │  salient moments + deltas
                                                    ▼
                                          veyo-enrich
                                            transcript (whisper.cpp) · caption salient frames ·
                                            OCR on-screen text · structure the event track
                                                    │  index
                                                    ▼
                              CloakPipe pass  ──►  strip PII / secrets from frames + transcript
                                                    │  clean index
                                                    ▼
                                          store  (the artifact: video + index)
                                                    │
                                                    ▼
                                          MCP server + JSON API  ──►  the URL an agent queries
```

**Hard rule:** the recorder owns *capture + cinematic + share handoff* and nothing else. Editor timelines, transitions, effect suites, analytics, and team admin are out (§8). "Rewrite Cap" means rebuild its *capture experience*, not its entire product surface — that's how the recorder stays thin while still being better. Full detail in [architecture.md](architecture.md).

**Stack note:** Rust for capture so it shares crates with veyo-core directly (no FFI, one toolchain, matches the wider stack). The cinematic/preview layer can render in a webview or natively. A small Rust web service handles share pages; Rust end-to-end keeps it single-toolchain. See [tech-stack.md](tech-stack.md).

## 4. Core features

Expanded, with acceptance criteria, in [features.md](features.md). In brief:

### 4.1 Quick, Loom-style sharing
Hit record, stop, get a link immediately. Upload-while-recording so the link is live the moment you stop. Local/self-host link by default; hosted option later (§7). The link resolves to a normal watchable video page *and* the agent-readable index behind the same URL.

### 4.2 Beautiful recording (the cinematic layer)
Cinematic auto-zoom that follows the cursor and clicks, smooth easing, dwell, anti-jitter. Backgrounds (gradients, images, blur), padding, rounded corners, shadows, device mockups (Safari/Chrome/Arc frames). Aspect-ratio presets. This makes a raw screen capture look produced with zero effort.

### 4.3 Agent-readable & processable recordings (the headline)
Every clip resolves, from one URL, to a structured object an agent explores **without downloading the video** — transcript, visual timeline (salient moments only, veyo-gated), OCR'd on-screen text, event track, summary + chapters, and metadata. Exposed via an **MCP server** and a **JSON API + per-clip `.json` sidecar**. Full shape in [index-schema.md](index-schema.md); query surface in [mcp-api.md](mcp-api.md).

> Design principle: *an agent should never need the pixels. If it does, the index failed.*

### 4.4 Process recordings that already exist (import-from-URL)
Paste a Loom / Cap / arbitrary video URL → run it through enrich → get the same index + MCP query, **no recording required.** Fastest path to traction, cheapest to build (zero capture code), and it immediately solves "stop sending me Looms, I can't pass them to an agent." Ship this *first* (see [phases.md](phases.md)).

### 4.5 Two capture backends, one schema
The agent doesn't care which backend produced the index; the schema is identical. See [capture-backends.md](capture-backends.md).

| Backend | Captures | Wins for | Effort |
|---|---|---|---|
| **browser** | DOM / a11y / console / network trace + sparse screenshots | web bug reports, feedback, QA flows | easy — ship first |
| **screen** | visual delta codec (veyo over pixels) + cursor/key events + cinematic layer | desktop demos, arbitrary apps | harder — realtime path |

### 4.6 CloakPipe in the pipe
Strip PII / secrets from frames + transcript before any index is shared. For browser mode, redact at the DOM level before screenshotting. A structural differentiator Cap / Loom / Builder's Clips can't match. See [privacy-and-redaction.md](privacy-and-redaction.md).

### 4.7 Searchable library
One index across every clip, dictation, and imported video. Full-text over transcripts + OCR'd on-screen text + captions (local SQLite + FTS).

## 5. What makes the recorder *thin, fast, useful*

- **Thin:** owns only capture + cinematic + share. No editor, no analytics, no team admin. Every feature that only improves the *human* recording experience and does nothing for the index is out of scope.
- **Fast:** Rust capture, CPU-side veyo gate, no pixel streaming. Enrichment runs async after the share link is already live — the user never waits on the agent index.
- **Useful from day one:** import-from-URL means it's valuable before a single line of capture code exists.

## 6. The headline demo

> Paste a clip link into Claude. Ask "what error showed up and what was the user doing right before it." Get an accurate, timestamped answer grounded in what was on screen and said — without Claude ever watching the video.

If that demo lands, the product is made. Everything else is plumbing to make that one interaction reliable.

## 7. Sharing model — the honest fork

- **Local / self-host (default, thesis-pure):** zero ongoing cost, no egress. Tunnel-style links (ngrok-class) work for *agent-on-my-machine* and *ephemeral peer view*; flaky for durable async human sharing (recipient needs your box up, links die).
- **Hosted cloud (optional, commercial, later):** durable links + human async sharing, but reintroduces video-hosting cost (storage + egress + transcode + CDN, forever).

Ship local as default; treat hosting as the open-core commercial layer.

## 8. Non-goals (holding the line)

- No full video editor (timelines, transitions, effects suite). That's how the recorder gets fat.
- No analytics / engagement dashboard (Loom's moat, irrelevant here).
- No team / workspace / SSO at v1. Single-user local first.
- No feature that improves only the human side and not the index.

## 9. Positioning vs the field

Full grounded analysis in [competitive-analysis.md](competitive-analysis.md). In short:

- **Loom** — human-only artifact, closed, now Atlassian-owned and getting pricier. The "can't pass this to an agent" pain is theirs.
- **Cap** — excellent recorder, local-first, self-host, AGPL-3.0. Closest on *capture*; not an agent-comprehension layer. clipxd matches the capture experience and adds the index. (Interop: import Cap recordings rather than fork its AGPL code.)
- **Builder's Clips (agent-native)** — same thesis, but a cloud serverless framework template, not local-first, no on-device codec, no redaction. clipxd is local-first, single-purpose, with veyo + CloakPipe underneath.
- **clipxd** — the only one where the index *is* the product and no raw imagery leaves the device.

## 10. Build phases

Detailed in [phases.md](phases.md). In short:

- **Phase 1 — import + enrich (no capture):** URL → enrich → JSON sidecar + MCP query. Smallest thing that proves the headline; supplies veyo-core its first real sessions.
- **Phase 2 — browser backend:** rrweb/Playwright-style trace → same schema. Covers the high-volume bug-report case.
- **Phase 3 — owned recorder:** Rust capture + cinematic layer → veyo pipeline → local share link.
- **Phase 4 — CloakPipe pass + hosted optional tier.**
- **Phase 5 — searchable library across everything.**
- **Phase 6 (optional) — cinematic auto-zoom as a standalone reusable crate.**

## 11. Licensing & IP notes

Full detail in [licensing.md](licensing.md).

- **Don't port Cap's source.** Cap is AGPL-3.0; a Go/Rust translation of it is an AGPL derivative and forecloses a closed hosted tier. Reimplement the *capture UX* clean-room instead. (Cap's `scap` and `cap-camera` crates are MIT and *are* reusable.)
- **Verify any cinematic-layer reference's license** before pulling its code; if permissive (MIT-class), reuse is fine, otherwise reimplement the zoom/background logic.
- **veyo stays Apache-2.0 core + commercial;** clipxd inherits the same open-core split.

## 12. Open questions

Tracked in [risks-and-open-questions.md](risks-and-open-questions.md).

- Is veyo-core hitting recall ≥ 0.9 at emission < 1% on ≥3 real sessions yet? Gates whether Phase 1 ships now or waits on the codec. (clipxd import can *generate* those sessions.)
- MCP: one server per clip, or one indexing the whole library?
- Recorder rendering: native cinematic layer, or webview?
- Does import-from-URL alone pull enough demand to justify the capture work before it ships?
- Auto-zoom: inside the recorder, or a standalone crate useful to others too?
